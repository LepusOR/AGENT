# PRD 理解回执

## 适用模式

- `light`：不使用；只做 1-3 句话目标复述
- `standard/full`：强制使用；用户 ack 前不得改代码

## 模板

```markdown
[prd-understanding]

## 业务目标
<一句话>

## 改动定位
- 页面：<PageName>（`src/pages/<PageName>/`）
- 元素：<具体组件 / 字段 / 列 / 筛选项>
- 类型：bug / feat

## 前后对照
> 「改动前」必须引用 `code_snapshot` 的真实代码事实。

| 维度 | 改动前（代码事实） | 改动后（PRD 目标） |
|---|---|---|
| <字段/状态/UI> | `<file:line>` `<≤ 5 行 snippet>` | <PRD 描述目标> |

## 验收标准（可测）
- [ ] <条目 1>
- [ ] <条目 2>

## 假设清单（必填，≥ 1 条）
1. <可被用户确认或否定的假设>

## 不确定项（如有）
- <需要用户回答的问题>

## 术语词典差量（full 必填）
<无变更时写「本次 PRD 未涉及术语新增或修订」>
```

## 用户确认

输出回执后停下问用户：

- 「全部正确，继续」（推荐）→ `user_ack = true`
- 「需要修正」→ 用户给出修正点，修订回执后再次确认
- 「中止」→ 终止本次 page-dev 流程

`user_ack !== true` 时，不得进入定位和改动。

## 假设清单指引

避免编造伪假设。优先从这些角度写：

- 字段语义、枚举值、可空性
- 旧数据兼容策略
- API / service 是否已存在
- 跨页面影响
- PRD 中文术语到代码实体的映射
- loading、空态、错误态等 UI 细节

每条假设必须能被用户明确确认或修正，不写「可能」「也许」。
